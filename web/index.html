System Overview

SailFrames is a complete sailboat racing data logging and analytics platform designed for competitive sailors who want detailed performance data without expensive commercial solutions.

Deployment

SailFrames is currently deployed at Courageous Sailing and Piers Park Sailing Center in Boston Harbor, Massachusetts. Six E1 fleet trackers (E1-E6) are used for J/80 and Sonar fleet racing.

Key Differentiators

  • PPK GNSS: First sailing platform using Post-Processed Kinematic positioning with free NOAA CORS data
  • Multi-sensor hardware: Dedicated IMU, wind, pressure sensors (not phone-only)
  • Fleet-wide logging: 6 boats on same course, same time, synchronized timestamps
  • Open source: Apache 2.0 license, full hardware and software stack
  • No permanent install: Daily install/remove in under 30 seconds

Data Captured

Every session records:

  • GPS: Position, SOG, COG, satellites, HDOP at 1Hz
  • Raw GNSS: RTCM3 binary observations for PPK post-processing
  • IMU: Heel, pitch, acceleration, gyro at 20Hz
  • Wind: AWA, AWS, TWA, TWS, TWD at 1Hz (E1 with Calypso sensor)

PPK GNSS Explained

Post-Processed Kinematic (PPK) positioning is what makes SailFrames unique. Instead of using real-time corrections (RTK), we log raw GNSS observations and process them after the race using base station data from NOAA's free CORS network.

How PPK Works

  1. Log raw observations: The LG290P GPS logs pseudorange and carrier phase measurements from all visible satellites
  2. Download CORS data: After sailing, download base station data from geodesy.noaa.gov/UFCORS (free, available ~1 hour after recording)
  3. Process in RTKLIB: Combine rover observations + base observations + navigation data to compute a fixed solution
  4. Result: Sub-meter accuracy without any base station hardware

Why PPK Instead of RTK?

Factor RTK PPK
Base station required Yes, on shore or boat No, uses NOAA CORS network
Real-time corrections Yes, via radio/cellular No, post-processed
Range from base <10km typical 50+km from CORS station
Connectivity Must maintain link None required on water
Accuracy 1-2cm (if link maintained) <1m (reliable)

RTKLIB Processing

To process raw observations:

# Load .rtcm3 directly in RTKPOST
# Configure: Positioning Mode = Kinematic, Frequencies = L1+L2+L5

PCB Design Files

Complete KiCad project available in edge-e/hardware/kicad_sailframes-e1/. Gerbers ready for JLCPCB fabrication (60.5 x 91.5mm, 2-layer, green solder mask).

Arduino IDE Setup

  1. Install ESP32 board support (v3.3.7+)
  2. Select: DOIT ESP32 DEVKIT V1
  3. Partition Scheme: Huge APP (3MB No OTA)
  4. Install libraries: TFT_eSPI, NimBLE-Arduino
  5. Configure TFT_eSPI User_Setup.h (see firmware docs)

Critical: GPIO12 Strapping Pin

Do NOT use GPIO12 for any external connection. GPIO12 controls flash voltage at boot - if pulled HIGH, ESP32 fails to start. Use GPIO35 (input-only) for SD MISO instead.

E1 Firmware Reference

The E1 firmware (sailframes_e1.ino) handles sensor acquisition, display updates, SD logging, and Wi-Fi upload.

Key Features

  • NMEA parsing (GGA/RMC/GSA/GSV from LG290P)
  • RTCM3 binary logging for PPK post-processing
  • BNO085 IMU at 20Hz (GAME_ROTATION_VECTOR mode)
  • TFT display with large speed/COG numbers
  • Battery monitoring via ADC
  • Wi-Fi auto-upload to S3 on known network detection
  • GPS speed-triggered auto-recording (starts >2kt, stops after 5min <0.5kt)
  • Calypso wind sensor BLE integration (E1 primary)

LG290P GPS Configuration

The firmware configures the LG290P at boot for NMEA + RTCM3 output:

// NMEA messages
sendPQTM("PQTMCFGMSGRATE,W,GGA,1");
sendPQTM("PQTMCFGMSGRATE,W,RMC,1");
sendPQTM("PQTMCFGMSGRATE,W,GSA,1");
sendPQTM("PQTMCFGMSGRATE,W,GSV,1");

// RTCM3 MSM7 for PPK
sendPQTM("PQTMCFGRTCM,W,7,0,-90,07,06,1,0");
sendPQTM("PQTMCFGMSGRATE,W,RTCM3-1019,1");  // GPS ephemeris
sendPQTM("PQTMCFGMSGRATE,W,RTCM3-1020,1");  // GLONASS ephemeris
sendPQTM("PQTMCFGMSGRATE,W,RTCM3-1042,1");  // BeiDou ephemeris
sendPQTM("PQTMCFGMSGRATE,W,RTCM3-1046,1");  // Galileo ephemeris

sendPQTM("PQTMSAVEPAR");  // Save to NVM
sendPQTM("PQTMHOT");      // Hot restart

Sensor Configuration

BNO085 IMU

Use GAME_ROTATION_VECTOR mode (6-DOF, no magnetometer) for heel and pitch on a boat. The magnetometer is unreliable due to lead keel, stainless rigging, and engine interference.

Calibration

  • Startup tare: Record current orientation as "zero" when mounted and boat is level
  • Gyro: Auto-zeros on startup (hold still 2-3 seconds)
  • Accelerometer: Gravity reference handles heel/pitch automatically

Common Issues

  • I2C errors: BNO085 requires 400kHz I2C due to clock stretching
  • 10-20 degree heading drift: Normal due to magnetometer interference - use GPS COG instead

LG290P GPS

The Waveshare LG290P is a quad-band (L1/L2/L5/L6) GNSS receiver capable of PPK post-processing. At ~$109 with antenna, it's the most affordable PPK-capable module available.

Configuration

Use QGNSS on Windows for full configuration (PyGPSClient has limited LG290P support).

$PQTMCFGRTCM,W,7,0,-90,07,06,1,0*     # Enable MSM7 for all constellations
$PQTMCFGMSGRATE,W,RTCM3-1019,1*       # GPS ephemeris
$PQTMCFGMSGRATE,W,RTCM3-1020,1*       # GLONASS ephemeris
$PQTMCFGMSGRATE,W,RTCM3-1042,1*       # BeiDou ephemeris
$PQTMCFGMSGRATE,W,RTCM3-1046,1*       # Galileo ephemeris
$PQTMCFGCNST,W,1,1,1,1,1,1*           # Enable all constellations
$PQTMSAVEPAR*                          # Save to NVM

Wind Sensor Integration

Current: Calypso Ultrasonic Mini

E1 currently uses the Calypso Ultrasonic Mini anemometer, which connects via Bluetooth LE 5.1.

Mounting

  • Mount the sensor with the bow-mark pointing forward
  • Use a suction cup mount + extension pole for no permanent modifications
  • Only one device can connect at a time (disconnect phone apps first)

AWA Correction

Known issue: The Calypso Mini reports AWA 180 degrees inverted from expected. SailFrames automatically applies a correction: (raw_awa + 180) % 360

In Development: Open Source Ultrasonic Wind Sensor

Design Goals

E1 Direct HTTP Upload

Due to ESP32 TLS limitations, E1 devices upload directly to S3 via HTTP (not HTTPS). This requires a bucket policy allowing unauthenticated PUT: 

{
    "Sid": "FleetDirectHTTPUpload",
    "Effect": "Allow",
    "Principal": "*",
    "Action": "s3:PutObject",
    "Resource": "arn:aws:s3:::sailframes-fleet-data-prod/raw/*"
}

Why HTTP? ESP32 Arduino Core 3.3.7 has broken TLS (mbedTLS BIGNUM allocation failure during RSA operations). Fleet data (GPS, IMU) is not sensitive.

Lambda Processing

S3 triggers Lambda on new uploads:

  1. Parse CSV files, validate data
  2. Merge multi-file sessions
  3. Calculate derived fields (TWA, TWS, TWD, VMG)
  4. Store processed JSON for web dashboard

Maneuver Detection

Tacks and gybes are detected using multiple signals:

  1. COG change: >60 degree heading change within 30 seconds
  2. IMU yaw rate: High turn rate confirms maneuver timing
  3. TWA flip: Sign change in TWA confirms tack vs gybe

VMG Calculation

# VMG = SOG * cos(angle to target)
# For upwind: target is TWD (wind direction)
# For downwind: target is TWD + 180

import math

def calc_vmg(sog, cog, twd, upwind=True):
    target = twd if upwind else (twd + 180) % 360
    angle_diff = abs(cog - target)
    if angle_diff > 180:
        angle_diff = 360 - angle_diff
    vmg = sog * math.cos(math.radians(angle_diff))
    return vmg

Troubleshooting

Common Issues

E1 Won't Boot After Wiring SD Card

Symptom: ESP32 fails to start, no serial output

Cause: GPIO12 (strapping pin) is pulled HIGH by SD card module

Fix: Use GPIO35 for SD MISO, not GPIO12

TFT Display Flickers During SD Writes

Symptom: Display blanks momentarily when logging to SD

Cause: TFT and SD sharing same SPI bus

Fix: Use separate buses - TFT on VSPI, SD on HSPI

E1 Wi-Fi Upload Fails with TLS Error

Symptom: "mbedTLS RSA BIGNUM memory allocation failed"

Cause: ESP32 Arduino Core 3.3.7 TLS is broken

Fix: Use HTTP direct to S3, not HTTPS. See E1_HTTP_UPLOAD_SETUP.md

Wind Sensor Shows Wrong Direction

Symptom: AWA reports wind from opposite side

Cause: Calypso Mini 180 degree inversion

Fix: Apply (raw_awa + 180) % 360 correction (already in firmware)

GPS Takes Forever to Get Fix

Symptom: 60+ seconds to first fix

Possible causes:

  • Cold start (device off >2-4 days): Normal, wait for fix
  • Poor sky view: Move antenna to better location
  • LG290P: SBAS helps, no A-GNSS service available

BLE Wind Sensor Won't Connect

Symptom: "Device not found" or connection timeout

Cause: Another device (phone app) is connected

Fix: Disconnect phone, only one BLE connection allowed
